# CS320
Working on this project reinforced how important it is to write software that is both functional and reliable from the very beginning. I learned that writing code is only one part of the development process; thoroughly testing that code is just as important. By creating comprehensive JUnit tests for the Contact, Task, and Appointment services, I was able to verify that each class behaved as expected while also ensuring that invalid data was rejected appropriately. Testing normal operation, boundary conditions, and invalid inputs gave me confidence that the application met its requirements and would behave predictably if it were used by real users. This project also demonstrated that writing validation into constructors and setter methods is an important step toward making software more secure because it prevents invalid or unexpected data from entering the system before it can cause larger issues.

Another important lesson from this project was learning how to interpret user requirements and translate them into working software. Rather than making assumptions about how the application should behave, I relied on the documented requirements to guide every design decision. Each requirement became part of both the implementation and the testing process. For example, limitations on field lengths, unique identifiers, and appointment dates were implemented directly in the code and then verified through unit tests. This approach ensured that the finished application reflected the customer's expectations instead of my own assumptions. It also showed me that successful software development begins with understanding what the user actually needs before writing any code.

My approach to designing software has become much more structured throughout this course. Instead of immediately focusing on implementation, I now begin by reviewing the requirements, identifying the data that needs to be stored, and determining how different classes should interact with one another. From there, I design each class with a single responsibility, implement the required functionality, and finally create unit tests to verify that each component works correctly before moving on. This project reinforced the value of developing software in small, manageable pieces and validating each step along the way. Moving forward, I plan to continue using this design process because it produces code that is easier to understand, easier to maintain, and significantly more reliable as projects become larger and more complex.
